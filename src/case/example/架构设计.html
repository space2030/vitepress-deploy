架构设计 #

  1. 关注需求
  2. 运用需求的结构化
  3. 必须超越《软件需求规格说明书》
  4. 尽早识别风险

01 需求是分层次的 #

  1. 业务级需求：包含客户或出资者要达到的业务目标、预期投资、工期要求，以及要符合那些标准，对哪些遗留系统进行整合等约束条件。
  2. 用户级需求：用户使用系统来辅助完成哪些工作？对质量有何要求？用户群及所处的使用环境方面有何特殊要求？
  3. 开发级需求：开发人员需要实现什么？开发期间、维护期间有何质量考虑？开发团队的哪些情况会反过来影响架构？ 从“需求定义了直接目标还是间接限制”的角度，把需求划分为3种类型，这就是需求的3个方面 功能需求：更多体现各级直接目标要求。 质量属性：运行期质量+开发期质量 约束需求：业务环境因素+使用环境因素+构建环境因素+技术环境因素 需求的结构绝对不是“List”而应该是“二维数组”

02 约束 #

1. 来自客户或出资方的约束性需求

  • a. 架构师必须充分考虑客户对上线时间的要求，预算限制，以及集成需要等非功能需求。
  • b. 客户所处的业务领域是什么？有什么业务规则和业务限制
  • c. 是否关注相应法律法规、专利限制
  • d. ......

2. 来自用户的约束性需求

  • a. 软件将提供给何阶层用户？
  • b. 用户的年龄段？使用偏好？
  • c. 用户是否遍及多个国家？
  • d. 使用期间的环境有电磁干扰、车船移动等因素吗？
  • e. ......

3. 来自开发者和升级维护人员的约束性需求 。

  • 如果开发团队的技术水平有限（有些软件企业甚至希望通过招聘便宜的程序员来降低成本）、磨合程度不高、分布在不同城市、会有何影响？
  • 开发管理方面、源代码保密方面、是否要顾及？

约束

  • 不能遗忘，业界当前技术环境本身也是约束性需求。
    • 技术平台，中间件，编程语言流行度，认同度，优缺点
    • 技术发展的趋势如何
  • 约束是架构设计的上下文，没有全局观念就不可能成为架构师
  • 软件需求 = 功能需求 + 质量属性 + 约束

03 关键质量目标 #

  1. 分类合适 + 必要扩充
    • 可管理性、开放性、仅有性能时补充效率，仅有效率时补充性能
    • 性能 = 速度 + 吞吐量 + 持续高速性（用户层面）
    • 效率 = CPU、内存、硬盘和网络的利用率（成本利用率层面）
  2. 考虑多方涉众
  3. 检查性思维
  4. 识别矛盾+划定优先级
  5. 严格程度符合领域与规模特点。任何时候，只有关键质量属性多于一个，都有考虑它们之间是否有矛盾关系，在第一时间做出权衡取舍，确定不同的优先级

04 功能需求 #

功能需求规定开发人员必须在产品中实现的软件功能，用户利用这些功能来完成任务，满足业务需求，功能需求有时也被称作行为需求。

05 关键功能 #

  1. 核心功能
  2. 必做功能
  3. 高风险功能
  4. 独特功能
  5. 必做功能
  6. 高风险功能
  7. 独特功能

06 实施 #

避免“架构 = 模块 + 接口”，在大型系统架构设计的早期比较注重识别重大需求，特色需求，高风险需求，据此来设计概念架构。

  1. 能否清晰地讲解概念架构，并借此说明“客户关心的价值如何实现、担心的问题如何解决”
  2. 概念性架构就是对系统设计的最初构想，就是把最关键的设计要素和交互的机制确定下来，然后考虑具体技术的运用，设计出实际架构。
  3. 概念架构满足“架构 = 组件 + 交互”的基本定义，只不过概念架构仅关注高层组件
  4. 概念架构对高层组件的“职责”进行了笼统的界定，并给出了高层组件之间的相互关系
  5. 概念架构不应涉及接口细节

07 现状 #

  1. 误将“概念架构”等同于“理想架构”
  2. 误将“阶段”当成“视图”

08 概念架构三步骤 #

  1. 初步设计，基于关键功能

  2. 高层分割，对系统进行高层切分

  3. 考虑非功能需求。

“架构”与“方案的关系”

概念架构难以支持并发开发，要支持开发组相对独立地进行工作，须提供指定和限制作用更明确的“规约”一级的设计。

  1. 细化架构和概念架构之间存在如下典型差异

细化

  • 接口，在细化架构中，接口占据非常核心的地位，而概念架构并不关系明确的接口定义，只有抽象的组件和抽象的交互机制

  • 子系统，细化架构重视通过子系统和模块来分割整个系统，并且子系统往往有明确的接口，而概念架构中只有抽象的组件，这些组件没有接口，只有职责，一般是处理组件、数据组件或连接组件的一种，当然概念架构中也有“大组件分解成小组件”的设计决策，但并非子系统的含义。

  • 交互机制，细化架构中的交互机制应是“实在的”，如基于接口编程、消息机制或远程方法调用等；

  • 而概念架构中的交互机制是“概念化”的，例如：“A层使用B层的服务”这里的“使用”到了细化架构中可能基于接口编程、消息机制、远程方法调用等其中的一种。

  • “方案”与架构的联系与区别: 方案包含一定的架构内容、方案涉及的架构基本在概念架构一级、架构设计的工作还远未完成

10 物理架构设计 #

  1. 物理架构视图着重考虑运行软件的计算机、网络、硬件设施等情况，还包括如何将软件包部署到这些硬件资源上，以及它们运行时的配置情况。
  2. 物理架构还要考虑软件系统和包括硬件在内的整个IT系统之间是如何相互影响的，由于一部分运行质量属性需要硬件或网络的支持，所有物理架构必须关注如何配置硬件和网络来满足软件系统的可靠性，可伸缩性，持续可以用性，性能，安全等方面的 要求。

11 运行架构设计 #

  1. 确定引入哪些控制流
  2. 确定每条控制流的任务
  3. 处理相关问题：控制流的创建，销毁，通信机制等等
  4. 进一步考虑：控制流之间的同步关系，若有资源争用还要引入加锁机制。控制流是一个处理机上顺序执行的动作系列，确定引入哪些控制流，并没有固定不变的套路。
