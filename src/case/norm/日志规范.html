日志规范 #

01 日志 #

  1. 日志，维基百科的定义是记录服务器等电脑设备或软件的运作。
  2. 日志文件提供精确的系统记录，根据日志最终定位到错误详情和根源

02 记录日志 #

  1. 系统初始化
  2. 编程语言提示异常
  3. 业务流程预期不符

03 日志规范 #

  1. if...else...判断是否正确的时候需要在else加入warn日志。
  2. 不打印无意义的日志。日志里面要带有业务信息，否则意义不大
// 错误示例
logger.Error("Consume message faild!!!")
// 正确示例
logger.Error=("Consume message faild, msgId{}".format(id))
  1. 不推荐使用字符串拼接，建议使用占位符。
  2. 循环体内不要打印Info日志
  3. 日志文件至少保存15天

04 日志命名 #

日志命名方式为appName-logType-logName.log

描述

  • appName：项目名称，如dwSpider, dwApi
  • logType：日志类别。推荐三个类型，stats:统计日志、monitor:诊断日志、vist:访问日志
  • logName：日志描述，可以为爬虫名称或者具体的日志描述
  • 如：dwSpider-monitor-baseUpdate.log

05 不多不少的日志 #

  1. 无用日志的常见情况。

    • 能够放在一条日志中的东西放在多条日志中输出
    • 预期会发生且能够正常处理的异常，打印一堆无用的堆栈
    • 为了开放调试方便而加入的临时日志

  2. 日志过少的情况

    • 请求出错时不能通过日志直接定位问题，需要添加临时日志并重新请求才能定位问题
    • 无法确定服务中的后台任务是否按照期望执行
    • 无法确定服务的内存数据结构的状态
    • 无法确定服务的异常处理逻辑是否正常
    • 无法确定服务器启动时配置是否正确加载

06 常见的会遗漏的日志 #

  1. 系统的配置参数。系统在启动过程中通常先读启动参数，可在系统启动后输出到日志中，方便确认系统是否安装期望的参数启动
  2. 后台定期执行的任务。如定期更新缓存的任务，可以记录任务开始时间，更新了多少缓存配置等，掌握定期执行任务的状态
  3. 异常处理逻辑。失败重试的日志也需要记录下来

07 不断优化的日志 #

  1. 日志不是一遍就能写好的，需要结合实际使用，问题的定位不断优化
  2. 如果定位问题时间过长，则说明需要优化
  3. 需要思考是否可通过优化日志来提前判断该问题是否可能发生
  4. 定义好团队的日志规范，保证每个开放记录的日志格式统一
  5. 定期对代码日志进行review

08 好的日志的特点 #

  1. 能够快速的定位问题
  2. 从日志就能了解问题的原因
  3. 了解线上系统的运行状态
  4. 发现系统的瓶颈
  5. 不能影响系统的正常运行

09 异常日志错误码 #

  1. 错误码的制定原则：快速溯源、沟通标准化。
    • 错误码必须能够快速知晓错误来源，可快速判断是谁的问题
    • 错误码必须能够进行清晰地比对
  2. 错误码不体现版本号和错误等级信息
  3. 全部正常，但不得不填充错误码时返回五个零：00000
  4. 错误码为字符串类型，共五位，分成两个部分：错误产生来源+四位数字编号。
    • 错误产生分为A/B/C 。A表示错误来源于用户，比如参数错误，用户安装版本过低，用户支付超时等问题； B表示来源于当前系统，往往是来源于当前系统，往往是业务逻辑出错，或程序健壮性差等问题； C表示错误来源于第三方服务，比如CDN服务出错，消息投递超时等问题；
    • 四位数字编号从0001到9999，大类之间的步长间距预留100
  5. 编号不与公司业务架构，更不与组织架构挂钩，以先到先得的原则在统一平台上进行，审批生效，编号即被永久固定。
  6. 错误码使用者避免随意定义新的错误码
  7. 错误码不能直接输出给用户作为提示信息使用

10 异常处理 #

  1. Java类库中定义的可以通过预检查方式规避的RuntimeException异常不应该通过catch的方式来处理，比如：NullPointerException，IndexOutOfBoundException等等。
  2. 异常捕获后不要用来做流程控制，条件控制。
  3. catch时请分清稳定代码和非稳定代码，稳定代码指的是无论如何不会出错的代码。对于非稳定代码的catch尽可能进行区分异常类型，再做对应得异常处理。
  4. 捕获异常是为了处理它，不要捕获了却什么都不处理而抛弃之，如果不想处理它，请将该异常抛给它的调用者，最外层的业务使用者，必须处理异常，将其转化为用户可以理解的内容。
  5. 事务场景中，抛出异常被catch后，如果需要回滚，一定要注意手动回滚事务。
  6. finally块必须对资源对象，流对象进行关闭，有异常也要做try-catch。
  7. 不要在finally块中使用return。try块中的return语句执行成功后，并不马上返回，而是继续执行finally块中的语句，如果此处存在return语句，则在此直接返回，会丢弃try块中的返回点
  8. 捕获异常与抛异常，必须是完全匹配，或者捕获异常是抛异常的父类。
  9. 在调用RPC、二方包、或动态生成类的相关方法时，捕获异常必须使用Throwable类来进行拦截。

11 日志规约 #

  1. 应用中不可直接使用日志系统中的API，而应依赖使用日志框架（SLF4J、JCL--Jakarta Commons Logging）中的API，使用门面模式的日志框架，有利于维护和各个类的日志处理方式统一。
// 使用 SLF4J
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
private static final Logger logger = LoggerFactory.getLogger(Test.class);
 
// 使用 JCL
import org.apache.commons.logging.Log;
import org.apache.commons.logging.LogFactory;
private static final Log log = LogFactory.getLog(Test.class);
  1. 所有日志文件至少保存15天，因为有些异常具备以“周”为频次发生的特点，对于当天日志，以“应用.log”来保存，保存在/home/admin/应用名/目录下，过往日志格式：{logname}.log.
  2. 根据国家法律，网络运行状态，网络安全事件，个人敏感信息操作等相关记录，留存的日志不少于六个月，并且进行网络多机备份。
  3. 应用中的扩展日志（如打点、临时监控、访问日志等）命名方式
  4. 在日志输出时，字符串之间的拼接使用占位符的方式。因为String字符串的拼接会使用StringBuilder的append（）方式，有一定的性能损耗。使用占位符仅是替换动作，可以有效提升性能。
  5. 对于trace/debug/info级别的日志输出，必须进行日志级别的开关判断
// 如果判断为真，那么可以输出 trace 和 debug 级别的日志
if (logger.isDebugEnabled()) {
   logger.debug("Current ID is: {} and name is: {}", id, getName());
}
  1. 避免重复打印日志，浪费磁盘空间，务必在日志配置文件中设置additivity=false
<logger name="com.taobao.dubbo.config" additivity="false" />
  1. 生产环境禁止直接使用 System.out 或 System.err 输入日志或使用 e.printStackTrace() 打印异常堆栈
  2. 异常信息应该包括两类信息：案发现场信息和异常堆栈信息，如果不处理，那么通过关键字 throws 往上抛出。
  3. 日志打印时禁止直接用 JSON 工具将对象转成 String
    • 如果对象里某些 get 方法被覆写，存在抛出异常的情况，则可能会因为打印日志而影响正常业务流程的执行
    • 打印日志仅打印业务相关属性值或者调用其对象的 toString()方法
